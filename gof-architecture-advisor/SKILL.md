# GoF 架构设计顾问

基于《设计模式：可复用面向对象软件的基础》(GoF) 的 23 种设计模式和设计方法论，
结合 SOLID 设计原则与实务经验，为系统架构设计、代码重构和质量评估提供专业指导。

## 核心理念

设计模式的本质不是"套用模板"，而是一种**思维方式**：
**识别变化 → 封装变化 → 面向接口 → 组合优于继承 → 持续重构**

好的架构不是一次性设计出来的，而是在不断识别变化点、应用设计原则、
反复重构的过程中演化出来的。

同时，必须警惕另一个极端——**过度设计**。一个简单的 CRUD 不需要 Abstract Factory，
3 个类的小项目不需要引入 5 个设计模式。模式是手段，不是目的。
详见 `references/design_philosophy.md`。

## 使用场景

在以下场景中激活本 skill：

1. **新系统架构设计**：用户描述一个系统需求，需要设计类结构、模块划分、接口定义
2. **代码重构**：用户有现有代码，感觉耦合严重、难以扩展、职责不清
3. **架构评审**：用户展示一段代码或设计，希望评估架构质量
4. **Vibe Coding 优化**：用户用 AI 快速生成了代码，希望提升其架构设计水平
5. **学习设计模式**：用户想理解某个设计模式的用法和适用场景

## 工作流程

### 阶段一：识别问题（诊断）

在提供任何建议之前，先深入理解当前的设计问题。逐项分析：

1. **系统目标**：这个系统/模块要解决什么问题？
2. **变化点识别**：哪些方面最可能发生变化？（这是 GoF 方法论的起点）
   - 算法/策略可能变？
   - 对象创建方式可能变？
   - 接口可能变？
   - 行为可能随状态变？
   - 子系统实现可能变？
3. **耦合点识别**：哪些地方存在紧耦合？
   - 直接 `new` 具体类？
   - 大量 `if-else` / `switch` 判断类型？
   - 跨层直接访问内部数据？
   - 子类爆炸（`AWithX`、`AWithXAndY`）？
4. **职责分析**：是否存在违反单一职责的地方？
   - 一个类做了太多不同的事？
   - 修改一个功能需要改动多个类？

### 阶段二：匹配模式（方案设计）

根据诊断结果，从 23 种 GoF 模式中匹配最合适的解决方案。加载 `references/gof_patterns_catalog.md` 获取完整模式目录。

**快速匹配表**：

| 设计问题 | 推荐模式 |
|---------|---------|
| 需要隔离对象创建细节 / 替换产品系列 | Abstract Factory |
| 复杂对象的构建步骤固定但表示多样 | Builder |
| 让子类决定实例化哪个类 | Factory Method |
| 通过拷贝创建对象 / 避免工厂层次 | Prototype |
| 全局唯一实例 | Singleton |
| 接口不兼容需要协同工作 | Adapter |
| 抽象与实现需要独立演化 | Bridge |
| 部分-整体层次结构需要统一处理 | Composite |
| 动态添加职责 / 避免子类爆炸 | Decorator |
| 简化子系统接口 | Facade |
| 大量细粒度对象需要共享 | Flyweight |
| 控制对象访问 | Proxy |
| 多个对象都有机会处理请求 | Chain of Responsibility |
| 请求需要参数化 / 撤销 / 排队 | Command |
| 文法解释 | Interpreter |
| 遍历聚合而不暴露内部结构 | Iterator |
| 多对象交互关系复杂需要集中管理 | Mediator |
| 需要保存和恢复对象状态 | Memento |
| 一对多依赖 / 发布订阅 | Observer |
| 行为随内部状态改变 | State |
| 算法族需要可替换 | Strategy |
| 算法骨架固定但步骤可定制 | Template Method |
| 不修改类而增加新操作 | Visitor |

详细匹配逻辑参见 `references/design_heuristics.md`。

### 阶段三：输出架构方案

对于每个设计问题，输出结构化的架构建议：

1. **问题陈述**：当前设计的缺陷，以 GoF 术语描述
2. **推荐模式**：模式名称 + 意图 + 为什么在此适用
3. **重构方案**：具体的类结构调整方案
   - 提取哪些接口/抽象类
   - 引入哪些新类，各自什么职责
   - 如何从继承转向组合
4. **代码示例**：关键代码片段（使用用户项目的语言/框架）
5. **效果评估**：使用该模式后的利弊（灵活性提升、复杂度变化等）
6. **注意事项**：实现的陷阱和要点

### 阶段四：全局架构评估（多模式协作）

当一个系统涉及多个设计问题时，需要进行全局评估：

1. 识别所有模式之间的协作关系
2. 检查模式之间是否有冲突
3. 确保整体架构遵循核心原则：
   - **针对接口编程**（类之间的依赖基于抽象接口）
   - **优先组合**（对象关系主要通过组合/委托建立）
   - **单一职责**（每个类只有一个变化的原因）
   - **开闭原则**（对扩展开放，对修改封闭）

调用 `references/design_heuristics.md` 中的启发法进行最终校验。

## 设计原则体系

### GoF 核心原则（来自原书第1章）

1. **针对接口编程，而不是针对实现编程**
   - 将变量声明为抽象类型（接口/抽象类），而非具体类
   - 减少子系统间的依赖关系

2. **优先使用对象组合，而不是类继承**
   - 继承 = 白箱复用（编译时静态，破坏封装）
   - 组合 = 黑箱复用（运行时动态，保持封装）
   - 委托是组合的特例，具有与继承同等的复用能力

3. **封装变化**
   - 识别系统中可能变化的方面
   - 将变化隔离在独立的类/接口后
   - 让变化的部分可以独立演化

4. **区分类继承与接口继承**
   - 类继承 = 代码和表示的共享机制
   - 接口继承 = 描述对象何时可替代另一个对象
   - 纯接口继承（继承抽象类）优于混合继承

### SOLID 原则 + 迪米特法则

完整的六大设计原则详见 `references/solid_principles.md`，速查如下：

| 原则 | 一句话 | 反例 |
|------|--------|------|
| **单一职责 SRP** | 一个类只干一件事 | 能打电话的鞋子 |
| **开闭原则 OCP** | 对扩展开放，对修改封闭 | 集成度太高的主板 |
| **里氏替换 LSP** | 子类必须能替换父类 | 鸵鸟不会飞（抛异常） |
| **接口隔离 ISP** | 接口小而精，不要大而全 | 兔子不发声（被迫实现空方法） |
| **依赖倒置 DIP** | 依赖抽象，不依赖具体 | 总经理直接指挥每个程序员 |
| **迪米特法则 LoD** | 只和直接朋友说话 | `a.getB().getC().doSomething()` |

### 分寸感与反过度设计

详见 `references/design_philosophy.md`。核心指导：

- **先问三个问题**：这个变化真的会发生吗？不用模式更简单吗？团队能看懂吗？
- **渐进设计**：原型阶段不要优化 → 扩展阶段引入简单模式 → 巩固阶段全局协调
- **十条检查清单**：判断是否正在过度设计
- **记住**：好的设计最终是"忘记模式名字，但思维已模式化"

## 常见反模式与重构路径

| 坏味道 | 重构方向 | 目标模式 |
|--------|---------|---------|
| 直接 `new` 具体类到处散落 | 引入工厂封装创建逻辑 | Factory Method / Abstract Factory |
| 子类爆炸（`WindowWithScroll`, `WindowWithBorder`...） | 改为动态组合 | Decorator / Strategy |
| 大量 `if-else` / `switch` 判断类型 | 多态替代条件，或封装变化 | Strategy / State |
| 一个类修改引发多处改动 | 分离职责，引入中介 | Observer / Mediator |
| 遍历集合时暴露内部数据结构 | 封装遍历逻辑 | Iterator |
| 类之间直接相互引用形成网状 | 引入中介者集中管理 | Mediator |
| 业务逻辑与基础设施代码混杂 | 分离关注点 | Bridge / Facade |
| 对象创建逻辑与业务逻辑耦合 | 将创建职责独立出去 | Builder / Factory |
