# 设计哲学：分寸、境界与反过度设计

> 来源：刘韬《秒懂设计模式》第25.7节"设计的最高境界" + GoF 第6章"结论"
> 定位：知道"不做什么"比知道"做什么"更重要

---

## 一、设计的三个层次

```
层次一：乱写（不知模式为何物）
    ↓ 学习设计模式
层次二：套用（知道了 23 个名字，见什么都想套）
    ↓ 内化、实践、踩坑
层次三：无招（忘记名字，但思维已经模式化）
```

**GoF 作者在 1994 年就说过**：
> "设计模式是人们可以更加简单方便地复用成功的设计和体系结构。……设计模式即使不能生成一个全新的应用，也有助于提高系统的可复用性。"
>
> 但他们也警告：
> "设计模式不能随意使用。……仅当你需要这种灵活性时才使用它。"

## 二、什么时候**不应该用**设计模式

### 2.1 三思而后行：先问自己三个问题

| 问题 | 如果答案是... | 行动 |
|------|-------------|------|
| 这个系统未来真的会在这个方面变化吗？ | "不太可能" | **不要用**——为不存在的灵活性付复杂度税 |
| 不用这个模式，现在的代码能读吗？ | "能，而且更简单" | **不要用**——简单是最大的美德 |
| 团队里其他人能看懂这个模式吗？ | "不确定" | **先用简单写法，注释里写 TODO 考虑 X 模式** |

### 2.2 典型场景：不需要设计模式

- **简单 CRUD 系统**：Controller → Service → Repository 已经够用，不需要 Abstract Factory 去创建 Repository
- **一次性脚本**：跑完就扔的代码，不值得花时间"架构"
- **原型验证**：先跑通再说。原型阶段用模式 = 在还不知道要不要转弯时先把方向盘锁死了
- **团队新人多**：用复杂的模式导致没人敢改代码，不如简单直白

### 2.3 不要用模式的十条检查清单

1. 系统只有 5 个以内的类？ → 不要滥用
2. 这段代码的下次修改时间线超过 6 个月？ → YAGNI（You Ain't Gonna Need It）
3. 引入模式后类数量翻倍但代码行数不降？ → 过度设计信号
4. 需要花 5 分钟向同事解释为什么用这个模式？ → 选更简单的
5. 模式只用在单个地方，没有复用可能？ → 过度抽象
6. 引入模式后需要写更多样板代码（boilerplate）？ → 算一下性价比
7. 接口只有一个实现类，且未来不会加第二个？ → 多余抽象
8. "万一将来..."开头的理由超过了 2 个？ → 停手
9. 实现这个模式用了超过 3 层的嵌套/转发？ → 比直接写的复杂度增加太多
10. 你本人一周后回来看这段代码，需要 30 秒以上才能看懂？ → 过度

## 三、"反过度设计"实战告警

### 3.1 典型走火入魔案例

**案例一：五类系统的三中介者**

```
UserService → UserValidationMediator → UserPersistenceMediator → UserNotificationMediator
```

每个 Mediator 协调 2 个对象。直接调用更清晰：
```
UserService → UserValidator
           → UserRepository
           → NotificationService
```

> 中介者只在对象间交互复杂（如 GUI 对话框中 5+ 个组件的联动）时才值得引入。

**案例二：CRUD 操作的 Strategy 堆砌**

```java
// 为了"可扩展的查询策略"
interface QueryStrategy { List<User> query(Query q); }
class ByNameStrategy implements QueryStrategy { ... }
class ByAgeStrategy implements QueryStrategy { ... }
// 结果：用户只有一个查询条件，而且是固定的
```

> 一个传参就解决的事，不要用 Strategy。Strategy 适用于算法族真正在运行时可替换的场景。

**案例三：所有类都先写接口**

```java
interface IUserService { ... }  // 只有一个实现类 UserServiceImpl
interface IOrderService { ... } // 只有一个实现类 OrderServiceImpl
```

> 接口是"协议"，不是"仪式"。一个实现类 + 没有多态需求 → 不需要接口。

### 3.2 过度设计的代价

| 代价 | 表现 |
|------|------|
| **认知负担** | 新人需要打开 10 个文件才能理解一个"用户登录" |
| **调试困难** | 多态 + 委托 + 工厂，F7 步进需要 20 步才能到核心逻辑 |
| **性能开销** | 每一层抽象的虚函数调用、对象创建、间接寻址都有成本 |
| **修改恐惧** | 接口一改就要连带改 5 个实现类 + Mock + 测试 |
| **维护噩梦** | 3 个月后原作者自己都看不懂为什么用了 Command 模式 |

## 四、设计的"渐进式"方法（GoF 推荐）

GoF 在第 2 章和第 6 章反复强调：**好的设计不是一次性想出来的**，而是通过"设计→审视→重构"的循环逐步演化的。

### 4.1 软件生命周期的三阶段模型

```
阶段一：原型（Prototype）
  → 快速实现，理解需求，不要过早优化
  → 此阶段允许"堆砌式编码"

阶段二：扩展（Expansion）
  → 需求变多，代码变复杂，"坏味道"开始出现
  → 开始引入设计模式应对具体痛点
  → 从最简单的模式开始（Factory Method → Strategy → ...）

阶段三：巩固（Consolidation）
  → 全局审视，统一抽象
  → 多个模式协作，形成一致的架构风格
  → 但要警惕"为了统一而统一"——不同抽象可能理应不同
```

### 4.2 模式引入的推荐顺序

| 场景 | 起步模式 | 为什么 |
|------|---------|--------|
| 创建对象不灵活 | Factory Method | 最简单，只需一个虚函数 |
| 需要在运行时替换行为 | Strategy | 把算法抽成一个接口+多个实现 |
| 需要给对象动态加功能 | Decorator | 比继承灵活，且可运行时组合 |
| 需要统一处理层次结构 | Composite | 递归+多态的自然延伸 |
| 需要解耦消息发送者和接收者 | Observer | 接口简单，概念直观 |

**不要从 Abstract Factory 起步**。它是最复杂的创建型模式，只在你已经清楚需要"产品系列"时才用。

### 4.3 从简单到复杂的演化路径

```
简单场景                         复杂场景
────────                        ────────
一个 if-else 分叉    →    Strategy（多种算法可替换）
直接 new 一个对象    →    Factory Method → Abstract Factory（多个产品系列）
一个类直接调用另一个  →    Observer → Mediator（多个对象复杂交互）
继承 3 个子类        →    Decorator / Composite（动态组合代替静态继承）
```

## 五、"无招"境界：从招式到内功

### 5.1 三个阶段的思维变化

| 阶段 | 状态 | 特征 |
|------|------|------|
| **学招式** | "我在用策略模式" | 知道每个模式的名字、结构、代码模板 |
| **用招式** | "这里应该封装变化" | 先看问题，再想方案，最后发现"原来这是策略模式" |
| **无招式** | "这段代码很自然" | 写的时候没想模式，但它天然满足 OCP、DIP、组合优于继承 |

### 5.2 判断你是否到了"无招"境界的三个信号

1. **设计评审时，你不再说"这里用 X 模式"**，而是说"这里的变化点是 Y，我们把 Y 抽出来封装成接口"
2. **看到别人的代码，你能一眼看出"这里如果加一个中间层就能解耦"**，但你不是在套模式——你是在识别变化点
3. **你写的代码没有显式的模式类名**（没有 Strategy、Factory、Observer 字样），但架构清晰、扩展容易

### 5.3 一句忠告

> "直到有一天，我们可能会忘记这些设计模式的名字，突破了'招式'和'套路'的牵绊，最终达到一种融会贯通的状态，各种'组合拳'信手拈来、运用自如。"
>
> ——刘韬《秒懂设计模式》第25.7节

## 六、Skill 的使用分寸

作为架构顾问，本 Skill 遵循以下原则：

1. **先听问题，再给建议**：不会一上来就推荐模式，先问"什么在变化？"
2. **从最简单的方案开始**：能用参数解决的问题不用工厂，能用工厂解决的问题不用抽象工厂
3. **明确告知代价**：每推荐一个模式，都说清楚它带来的复杂度和适用前提
4. **支持"不"**：当判断用户的场景不需要某模式时，果断说"这里保持简单就好"
5. **鼓励迭代**：建议先写简单版本，等真正痛了再重构引入模式

> 记住 GoF 原文的警告：
> **"设计模式不能随意使用。通常你通过引入额外的间接层次获得灵活性和可变性的同时，也使设计变得更复杂和/或牺牲了一定的性能。一个设计模式只有当它提供的灵活性是真正需要的时候，才有必要使用。"**
