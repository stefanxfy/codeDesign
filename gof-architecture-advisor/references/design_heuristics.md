# GoF 设计模式 — 设计启发法与实践方法论

> 从《设计模式：可复用面向对象软件的基础》第2章（实例研究：Lexi文档编辑器）和第6章（结论）中提炼的设计思维方法、启发法则和实践指导。

---

## 一、实践设计方法论：思考过程

### 1.1 问题驱动：先明确问题，再寻找解法

GoF 的设计方法论并非 "先有模式再套用"，而是 **从具体设计问题出发**。第2章开篇就列出了 Lexi 设计中的 7 个具体问题：

| 问题 | 对应模式 |
|------|----------|
| 文档结构 | Composite |
| 格式化 | Strategy |
| 修饰用户界面 | Decorator |
| 支持多种视感标准 | Abstract Factory |
| 支持多种窗口系统 | Bridge |
| 用户操作（含撤销/重做） | Command |
| 拼写检查和断字处理 | Iterator + Visitor |

**核心方法论**：对每个问题，先陈述 **目标** 和 **约束条件**，然后探索多种方案，评估权衡，最终找到合适的模式。模式是解决方案的结果，而非起点。

### 1.2 权衡驱动的决策过程

书中对每个设计选择都进行了显式的权衡分析：

- **格式化算法**：格式化质量 vs. 响应速度 vs. 存储需求。用户可能在不同场景下偏好不同，因此需要运行时可替换的算法。
- **窗口系统抽象**：功能交集（最小公共接口，能力受限）vs. 功能并集（接口庞大不一致，维护困难）→ 折中方案：提供 "支持大多数窗口系统的方便接口"。
- **遍历与分析的分离**：把分析放在 Iterator 中（紧耦合）vs. 放在 Glyph 类中（接口膨胀）vs. 独立 Visitor 对象（灵活但增加类层次）。

### 1.3 从具体到抽象，识别重复模式

GoF 的设计过程展示了一种 "具象→抽象" 的思维路径：

1. 先在具体上下文中解决问题（Lexi 的文档编辑器）
2. 然后退一步观察：这个问题类别在其他系统中是否也会出现？
3. 将解决方案提炼为可复用的模式

例如：Decorator 模式的 "透明围栏"（transparent enclosure）概念是这样提炼的——"我们根据这些得出了透明围栏的概念，它结合了两个概念：① 单子女（单组件）组合；② 兼容的接口。"

## 二、核心设计启发法

### 启发法 #1：封装变化（Encapsulate What Varies）

**这是贯穿全书的黄金法则。** GoF 对每个设计问题都问同一个问题：**"什么在变化？"** 然后将变化的部分封装起来。

| 变化的概念 | 封装方式 | 模式 |
|-----------|---------|------|
| 格式化算法 | 封装在 Compositor 对象中 | Strategy |
| 用户界面修饰 | 封装在透明围栏（Decorator）中 | Decorator |
| 窗口组件创建 | 封装在工厂对象中 | Abstract Factory |
| 窗口系统实现 | 封装在 WindowImp 层次中 | Bridge |
| 用户请求 | 封装在 Command 对象中 | Command |
| 遍历方式 | 封装在 Iterator 对象中 | Iterator |
| 分析操作 | 封装在 Visitor 对象中 | Visitor |

**原文引用**："对变化的概念进行封装。"（2.6.2 节）"再一次强调，一个好的解决方案是封装那些变化的概念。"（2.8.2 节）"这里是另一个封装变化概念的例子，即封装请求。"（2.7.1 节）

### 启发法 #2：组合优于继承（Favor Composition Over Inheritance）

GoF 反复通过 "子类爆炸" 的反面案例来说明为什么对象组合更好：

- **修饰用户界面**：用继承 → `BorderedScrollableComposition` 等类爆炸；用组合 → 透明围栏（Decorator）
- **窗口系统**：用继承 → 每个 Window 子类都要有对应每个平台的版本，`n × m` 个子类；用组合 → Bridge 模式，两个独立演化的类层次
- **用户操作**：用继承 → "所生成的类的数目就是窗口组件类型的数目和请求数的乘积"

**启发**：当你发现自己在创建 `AWithX`、`BWithX`、`AWithXAndY` 这种组合子类时，应该停下来考虑对象组合。

### 启发法 #3：面向接口编程（Program to an Interface）

一致性地对待对象是 GoF 设计的核心原则：

- "我们应该一致地对待文本和图形……避免将图形看作文本的一种特殊情形，或将文本看作图形的特例"（2.2 节）
- "我们的实现不应该过分强调内部表示中单个元素和元素组之间的差别"（2.2 节）
- Glyph 接口使得行、列、字符、图像都可以统一处理
- Command 接口使得所有用户操作（剪切、粘贴、改变字体、打印）通过同一个 Execute 方法执行

### 启发法 #4：分离关注点（Separate Concerns）

GoF 多次展示了将紧密耦合的职责分离的决策：

- **表示 vs. 格式化**："表示和格式化是不同的。记录文档物理结构的能力并没有告诉我们怎样得到一个特殊的格式化结构。"（2.3 节）→ Composite + Strategy
- **遍历 vs. 动作**："如果我们能区别遍历和遍历过程中所执行动作之间的差别，就可以得到更多的灵活性和潜在复用性"（2.8.5 节）→ Iterator + Visitor
- **窗口逻辑 vs. 窗口实现**："Window 类接口是针对应用程序员的，而 WindowImp 接口是针对窗口系统的"（2.6.4 节）→ Bridge

### 启发法 #5：运行时可配置优于编译时硬编码

- 格式化算法：通过 `SetCompositor` 在运行时改变
- 视感标准：通过替换工厂对象在运行时改变
- 窗口系统：通过替换 WindowImp 在运行时改变
- 用户操作：通过给 MenuItem 配置不同的 Command 对象

### 启发法 #6：识别稳定的部分和不稳定的部分

Visitor 模式的分析提供了重要洞察：

> "在使用 Visitor 模式之前你要问自己的一个重要问题是：哪一个类层次变化得最厉害？"（2.8.8 节）

在 Lexi 中，Glyph 类层次相对稳定（字符、行、列、图像等基本不会变），而分析操作（拼写检查、语法检查、字数统计等）会持续增加。因此 Visitor 是合适的。**反之，如果对象结构频繁变化而操作稳定，则不应使用 Visitor。**

### 启发法 #7：从简单开始，逐步引入抽象

Lexi 的设计过程展示了渐进式抽象：

1. 先用直接方式表达问题（如递归组合 → Glyph 接口）
2. 当发现维护困难或扩展受限时，引入新的抽象层（如 Monoglyph → Decorator）
3. 当同一模式在不同问题中出现时，提炼为通用模式

## 三、模式在真实项目中的应用指南

### 3.1 何时引入模式

GoF 在第2章中暗示了引入模式的时机判断：

- **当你在多个地方写相同或类似的构造器调用时** → Abstract Factory
- **当你发现自己在创建 "带有 X 和 Y 的 A" 这种组合子类时** → Decorator
- **当你需要支持多种算法且不希望到处散落条件判断时** → Strategy
- **当你需要在不修改已有类的情况下增加新功能时** → Visitor
- **当你需要支持撤销操作时** → Command
- **当客户代码需要遍历复杂结构但不关心内部细节时** → Iterator

### 3.2 模式不局限于特定领域

GoF 强调模式的通用性（2.9 节）：

- 财务分析应用：Composite 表示投资组合
- 编译器：Strategy 处理不同目标机的寄存器分配
- 任何 GUI 应用：Decorator 和 Command

### 3.3 模式之间的协作与镶嵌

> "设计良好的面向对象系统通常有多个模式镶嵌在其中"（第5章结论）
> "最佳的设计要用到许多设计模式，它们契合交织，形成一个更大的整体"（6.5 节）

Lexi 中的模式协作示例：
- Composite 定义的文档结构被 Iterator 和 Visitor 遍历和分析
- Abstract Factory 创建的产品被 Bridge 的 Window 类使用
- Command 操作的对象是 Composite 结构中的 Glyph
- Decorator 修饰的是 Composite 中的 Composition 对象

### 3.4 模式选择的关键问题

使用模式前应问自己（综合各章）：

1. **什么在变化？什么保持稳定？**
2. **哪一部分变化得更频繁？**
3. **这个变化需要在编译时还是运行时处理？**
4. **引入这个模式会增加还是减少整体复杂度？**
5. **模式之间的接口是否足够通用以支持未来的扩展？**

## 四、通用设计词汇与沟通价值

### 4.1 模式的本质是命名和分类

> "计算机科学家对算法和数据结构进行命名和分类，但我们却很少为其他类型的模式命名。"（6.1.1 节）

设计模式的核心贡献不是发明新技术，而是 **为已有的最佳实践命名**。这使得：

- 你可以说 "这里使用 Observer 模式" 而不是花 10 分钟解释一套发布-订阅机制
- 你可以说 "让我们从这些类中抽出一个 Strategy" 来表达一个清晰的设计意图
- 团队讨论的层次从代码实现提升到设计结构

### 4.2 模式作为学习和理解的工具

> "大多数规模较大的面向对象系统都使用了这些设计模式。学习这些设计模式将有助于你理解已有的面向对象系统。"（6.1.2 节）

> "按照一个系统所使用的设计模式来描述该系统可以使他人理解起来容易得多，否则，就必须对该系统的设计进行逆向工程来弄清楚其使用的设计模式。"

### 4.3 模式作为文档的辅助手段

> "设计模式使你可以更多地描述'为什么'这样设计而不仅仅是记录你的设计结果。"（6.1.3 节）

模式提供了一种 "设计意图文档化" 的方式——不仅记录了设计结果（是什么），还记录了设计理由（为什么）。

## 五、重构指导

### 5.1 软件生命周期的演化模型

GoF 引用了 Brian Foote 的软件生命周期三阶段模型（6.1.4 节）：

```
原型阶段 (Prototype) → 扩展阶段 (Expansion) → 巩固阶段 (Consolidation)
    ↑                                                |
    └──────────────────── 循环 ───────────────────────┘
```

1. **原型阶段**：快速搭建，类层次直接反映问题域实体，主要通过继承（白箱复用）
2. **扩展阶段**：不断满足新需求，软件变得臃肿、僵化
3. **巩固阶段**：重构不可避免——拆分职责、提升/下放操作、合理化接口。黑箱复用（对象组合）代替白箱复用（继承）

### 5.2 模式作为重构的目标

> "我们的设计模式记录了许多重构产生的设计结构。在设计初期使用这些模式可以防止以后的重构。不过你即使是在系统建成以后才了解如何使用这些模式，它们也可以教你如何修改你的系统。**设计模式为你的重构提供了目标。**"（6.1.4 节）

换句话说：如果不知道要把代码重构成什么样，模式提供了清晰的 **目标形态**。

### 5.3 预防性设计与响应式重构

GoF 主张两者结合：

- **预防性**：深入分析需求，识别易于变化的部分，在初期设计中使用模式保护这些变化点
- **响应式**："好的设计者不仅知道哪些变化会促使重构，而且还知道哪些类和对象结构能够避免重构——其设计对于需求变化具有健壮性。"

### 5.4 重构的常见方向

从 Lexi 案例中可归纳出典型的 "重构方向"：

| 重构前 | 重构后 | 动机 |
|--------|--------|------|
| 硬编码的构造器调用 | 通过工厂创建（Abstract Factory） | 支持产品系列的替换 |
| 继承实现的变体 | 组合实现的变体（Strategy） | 运行时替换算法 |
| 子类爆炸 | Decorator 装饰 | 动态组合功能 |
| 将分析操作放进类中 | 独立 Visitor | 不修改已有类即可增加新操作 |
| 直接内嵌遍历逻辑 | Iterator 封装 | 统一遍历接口 |
| 请求直接调用 | Command 封装 | 支持撤销/重做/排队 |

## 六、核心语录

> "对变化的概念进行封装。" — 全书最核心的设计法则

> "哪一个类层次变化得最厉害？" — 选择模式的关键问题

> "了解做什么要比为什么容易，而一个模式的'为什么'就是它要解决的问题。" — 理解模式本质

> "最佳的设计要用到许多设计模式，它们契合交织，形成一个更大的整体。以一种松散的方式把一些模式串接在一起来建造建筑是可能的。这样的建筑仅仅是一些模式的堆砌，而不紧凑。这不够深刻。然而另有一种组合模式的方式，许多模式重叠在同一个物理空间里：这样的建筑非常紧凑，在一小块空间里集成了许多内涵；由于这种紧凑，它变得深刻。" — Christopher Alexander，引自 6.5 节

> "设计模式为你的重构提供了目标。" — 6.1.4 节

## 七、总结：GoF 设计方法学的本质

GoF 的设计方法学可以概括为：

1. **从问题出发**：识别具体的设计问题、目标和约束
2. **识别变化**：找出系统中哪些方面最可能变化
3. **封装变化**：用抽象接口和对象组合隔离变化
4. **权衡评估**：明确每个设计选择的利弊
5. **复用经验**：将成功的设计结构提炼为可命名的模式
6. **持续重构**：将设计逐步推向更通用、更灵活的形态

这不是一套僵化的步骤，而是一种 **思维方式**——在面对任何设计问题时，都可以运用这套思维工具来寻找优雅的解决方案。
