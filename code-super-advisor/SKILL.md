# Code Super Advisor

代码设计超级顾问 — 一个**编排型** skill。

本身不提供具体的架构/模式/重构知识，而是根据用户输入判断工作模式，自顶向下调度子 skill 的知识库完成全层次审查或设计，最终输出一份结构化文档。

---

## 三大职责

| 职责 | 输入 | 输出 |
|------|------|------|
| **审查代码** | 代码文件 / 代码片段 / 代码库路径 | 代码审查报告 |
| **审查实现方案** | 设计文档 / 架构方案 / 技术方案 | 方案评审报告 |
| **辅助设计** | 需求描述 / 问题陈述 | 设计建议书 |

---

## 工作流程

### 第一步：判断工作模式

根据用户输入判断：

| 输入特征 | 工作模式 |
|---------|---------|
| 包含代码文件、代码片段、代码库路径 | 审查代码 |
| 包含设计文档、方案描述、架构图、技术选型 | 审查实现方案 |
| 包含需求描述、问题陈述、功能描述 | 辅助设计 |
| 混合输入（既有代码又有方案） | 先审查方案，再审查代码 |

如果模式不明确，直接询问用户。

### 第二步：调度子 skill（自顶向下）

无论哪种模式，都按**系统层 → 模式层 → 代码层**的顺序调用子 skill：

#### 1. 系统层 — clean-architecture-advisor

激活 `clean-architecture-advisor` skill，参考其 `references/` 下的文件：

| 模式 | 关注点 |
|------|--------|
| 审查代码 | 组件边界是否清晰？依赖方向是否正确？分层是否合理？是否有「尖叫的架构」？可选项是否保留？ |
| 审查方案 | 架构设计是否合理？边界划分是否恰当？策略层次是否正确？是否过度设计？ |
| 辅助设计 | 如何划分组件？如何画边界？依赖规则如何落地？部署模式如何选择？ |

参考文件：
- `clean_architecture_principles.md` — 核心原则（依赖规则、尖叫的架构、保留可选项、测试边界等）
- `component_principles.md` — 组件聚合/耦合原则（REP/CCP/CRP/ADP/SDP/SAP + 主序列量化）
- `architecture_patterns.md` — 边界形态、解耦模式、服务组件化、打包方式对比

#### 2. 模式层 — gof-architecture-advisor

激活 `gof-architecture-advisor` skill，参考其 `references/` 下的文件：

| 模式 | 关注点 |
|------|--------|
| 审查代码 | SOLID 原则是否违反？类关系是否过度耦合？模式是否滥用或缺失？ |
| 审查方案 | 模式选择是否恰当？是否过度设计？类职责划分是否合理？ |
| 辅助设计 | 推荐哪些设计模式？类关系如何组织？SOLID 如何落地？ |

参考文件：
- `gof_patterns_catalog.md` — 23 种设计模式目录（含实务推荐）
- `solid_principles.md` — SOLID 原则 + 迪米特法则
- `design_heuristics.md` — 设计启发法 + 重构指导
- `design_philosophy.md` — 过度设计警告 + 分寸感
- `oop_basics.md` — OOP 基础

#### 3. 代码层 — refactoring-advisor

激活 `refactoring-advisor` skill，参考其 `references/` 下的文件：

| 模式 | 关注点 |
|------|--------|
| 审查代码 | 22 种坏味道诊断 → 重构手法推荐 → 优先级排序 |
| 审查方案 | 预判可能的坏味道（过长函数？过大类？数据泥团？）→ 可维护性预评估 |
| 辅助设计 | 代码结构建议（如何避免常见坏味道？推荐什么代码组织方式？） |

参考文件：
- `bad_smells.md` — 22 种坏味道诊断指南
- `refactoring_catalog.md` — 72 种重构手法目录 + 决策树
- `refactoring_methodology.md` — 重构方法论
- `refactoring_example.md` — Movie Rental 完整示例
- `skill_coordination.md` — 三 skill 协同指南

### 第三步：汇总输出

将三个子 skill 的分析结果汇总为一份结构化文档。**无论哪种模式，最终都必须输出文档**，而非零散的聊天回复。

文档模板详见 `references/report_templates.md`。

## 编排规则

### 规则 1：自顶向下

始终按 **系统层 → 模式层 → 代码层** 的顺序分析。原因：
- 架构问题的影响范围最大，应优先发现
- 模式问题往往是架构问题的表征
- 代码坏味道可能是架构/模式问题的下游症状

### 规则 2：问题归因

发现问题时，尝试追溯根因：
- 代码坏味道 ← 可能是模式选择不当 ← 可能是架构边界不清
- 在报告中标注因果链，而非孤立罗列问题

### 规则 3：严重度分级

所有发现按以下标准分级：

| 级别 | 含义 | 处理时机 |
|------|------|---------|
| **P0** | 阻断性问题 — 架构方向错误、核心逻辑缺陷 | 立即修复 |
| **P1** | 重要问题 — SOLID 违规、严重坏味道、边界模糊 | 短期修复（1-2 周） |
| **P2** | 改进建议 — 可优化的模式选择、轻微坏味道 | 中期优化（1 个月） |
| **P3** | 长期建议 — 架构演进方向、技术债规划 | 长期改善 |

### 规则 4：聚焦原则

- 如果代码量很小（单文件/单函数），可以跳过架构层，直接从模式层或代码层开始
- 如果是纯架构设计（无代码），可以跳过代码层
- 不要为了"走完全流程"而强行分析不相关的层次

### 规则 5：文档优先

- 分析过程中可以向用户提问澄清，但最终交付物必须是文档
- 文档使用 Markdown 格式
- 文档中每个问题都应包含：位置、描述、严重度、建议

## 与子 Skill 的关系

```
用户请求
    │
    ▼
┌─────────────────────┐
│  code-super-advisor  │  ← 编排者：判断模式、调度、汇总
│  (本 Skill)          │
└──────┬──────────────┘
       │
  ┌────┼────────────────┐
  │    │                │
  ▼    ▼                ▼
┌──────┐ ┌──────────┐ ┌──────────────┐
│clean │ │   gof    │ │refactoring   │
│arch  │ │  arch    │ │  advisor     │
│adv   │ │  adv     │ │              │
└──────┘ └──────────┘ └──────────────┘
 系统层    模式层       代码层
```

**code-super-advisor 不替代子 skill**。当用户的请求明确只涉及单一层次时（如"帮我重构这个函数"），应直接激活对应的子 skill，而非通过本 skill 绕一圈。

本 skill 的价值在于：当用户需要**全层次、系统性**的审查或设计时，提供一站式编排。
