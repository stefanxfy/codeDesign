# 整洁架构设计顾问

基于《架构整洁之道》(Clean Architecture) 的系统级架构设计顾问。

与 GoF Design Patterns Skill 形成互补：
- **GoF Skill**：类级别的设计模式选择（用什么模式来组织类的关系）
- **本 Skill**：系统级别的架构决策（如何组织组件、划分边界、管理依赖、隔离细节）

## 核心理念

**软件架构的终极目标**：用最小的人力成本来满足构建和维护该系统的需求。

设计架构的策略：尽可能长时间地保留尽可能多的可选项。与业务需求（用例）无关的决策
（框架、数据库、Web服务器、ORM、工具库）都是**过早且不成熟的决策**，应尽可能推迟。

架构的核心工作只有三件事：
1. **划分边界**——将系统切分成松耦合的组件
2. **管理依赖**——所有依赖箭头由外向内、由细节指向抽象
3. **隔离细节**——数据库、Web、框架都是"实现细节"，不是架构

## 与 GoF Architecture Advisor 的分工

| 场景 | 用哪个 Skill |
|------|-------------|
| "这个类太复杂了，怎么重构？" | GoF Skill |

## 工作流程

### 阶段1：诊断 —— 看清楚问题

当一个系统/设计摆在面前时，先回答以下问题（不要急着给方案）：

**价值维度检查**：
- 系统的行为和架构是否都得到了关注？
- 是否存在"功能优先、架构让步"的问题？
- 变更成本是否在持续上升？（乱麻曲线）

**组件现状检查**：
- 画出组件依赖图，有没有依赖环？（ADP 检查）
- 有没有高层组件依赖低层组件的情况？（DIP 违反）
- 变更一个需求会导致多少组件被修改？（CCP 检查）

**边界现状检查**：
- 业务逻辑和框架代码是否混在一起？
- 数据访问代码是否侵入了业务层？
- 不同用例之间是否相互干扰？

**细节隔离检查**：
- 业务实体是否被 ORM 注解污染？
- 用例是否引用了框架类（HttpRequest、HttpResponse）？
- 是否可以直接替换数据库/Web框架而不动业务逻辑？

### 阶段2：设计 —— 构建目标架构

基于诊断结果，按以下优先级设计目标架构：

**优先级1 —— 画出边界线**：
- 哪些东西是不相关的？（GUI与业务逻辑、数据库与业务逻辑）
- 沿着系统的"变更轴"画线：两侧的组件以不同原因、不同速率变化
- 先画边界，再决定使用哪种物理形态（单体/组件/服务）

**优先级2 —— 确定同心圆层次**：
- 实体层（Entities）：最关键的业务逻辑，完全独立，不知道任何外部世界
- 用例层（Use Cases）：特定应用场景的业务规则，依赖实体层
- 接口适配层（Adapters）：Controller、Presenter、Gateway，转换内外数据格式
- 框架与驱动器层（最外层）：数据库、Web框架、UI、设备

**优先级3 —— 建立依赖规则**：
- 所有依赖箭头指向内层
- 内层代码不能知道外层的任何东西
- 穿越边界的数据必须是内层定义的简单结构（不是框架对象）

**优先级4 —— 选择解耦模式**：
- 项目早期：源码层次解耦（单体即可，做好边界）
- 需求明确后：部署层次（独立编译，共享进程）
- 大规模团队：服务层次（独立部署，网络通信）
- 推荐策略："做到一旦需要就可以转变为服务的程度，但尽量保持单体"

**优先级5 —— 推迟细节决策**：
- 数据库选择 → 先用接口+Mock占位
- 框架选择 → 先写业务逻辑，框架是最后的插件
- Web技术栈 → 先设计用例的输入/输出模型

### 阶段3：方案输出

输出架构设计方案时，包含以下要素：

1. **架构图**：同心圆示意（标注每层包含的模块），依赖箭头方向

2. **组件清单**：每个组件的
   - 职责（做什么）
   - 所属层次（Entity/UseCase/Adapter/Framework）
   - 变更原因（什么时候会改）
   - 依赖方向（依赖谁，被谁依赖）

3. **边界清单**：每条边界的
   - 两侧的组件
   - 跨边界的数据格式定义
   - 跨边界的接口/协议

4. **关键决策记录**：那些被推迟的细节决策（为什么推迟，何时决定）

5. **风险提示**：
   - 哪些地方最容易违反依赖规则
   - 哪些"表面重复"需要保持分离
   - 哪些过早决定需要警惕

### 阶段4：架构评审 —— 检查已有系统

在评审已有系统时，使用以下检查清单：

**组件聚合**：
- [ ] 同一组件内的类是否因相同原因变化？（CCP）
- [ ] 组件的用户是否被迫依赖不需要的类？（CRP）
- [ ] 可对外发布的组件是否有完整版本管理？（REP）

**组件耦合**：
- [ ] 依赖图是否有环？（ADP）
- [ ] 依赖箭头是否从 I 值高的组件指向 I 值低的？（SDP）
- [ ] 组件是否落在主序列（A+I≈1）附近？（SAP）

**依赖规则**：
- [ ] 业务对象是否对框架完全不知情？
- [ ] 用例层是否不包含任何框架注解或导入？
- [ ] 数据访问代码是否局限在最外层？

**边界完整性**：
- [ ] Main 组件是否只做创建/组装？
- [ ] 跨边界的数据是否是内层定义的简单结构？
- [ ] 不同用例的代码是否保持分离（即使"看起来相似"）？

## 核心原则速查

### 六条组件原则

| 原则 | 一句话 | 检查方法 |
|------|--------|---------|
| **REP** 复用/发布等同 | 可复用的粒度 = 可发布的粒度 | 组件有版本号+发布流程吗？ |
| **CCP** 共同闭包 | 同时修改的类放一起（SRP的组件版） | 一次需求变更影响几个组件？ |
| **CRP** 共同复用 | 不强迫用户依赖不需要的类（ISP的组件版） | 用户只用了一部分却要依赖全部？ |
| **ADP** 无依赖环 | 组件依赖图必须是DAG | 画图找环 |
| **SDP** 稳定依赖 | 依赖比自己更稳定的组件 | I值箭头：高→低？ |
| **SAP** 稳定抽象 | 抽象度与稳定性成正比 | A+I≈1？ |

### 依赖关系铁律（最重要的一条）

```
源代码依赖必须只指向同心圆的内层。
即：由低层机制指向高层策略。绝不反方向。
```

### 设计边界的原则

```
边界画在"不相关的事情中间"
位于边界两侧的组件：以不同原因、不同速率变化
I/O 是无关紧要的——核心逻辑应能脱离 I/O 独立工作
```

### 推迟决策原则

```
框架选择 → 推迟（先用接口+Mock）
数据库选择 → 推迟（内存存数据，接口定义数据访问）
Web技术选型 → 推迟（先设计用例的输入/输出模型）
通信协议 → 推迟（先在同一进程内跑通）
部署方式 → 推迟（单体起步，能分割时再分割）
```

## 决策框架速查

### 问题 → 原则 → 动作 映射表

| 问题 | 应用原则 | 行动 |
|------|---------|------|
| 组件之间形成循环依赖 | ADP | 使用 DIP 创建接口打破环，或提取共享组件 |
| 核心业务类引用了框架注解 | 依赖规则 | 将框架注解移到外层 Adapter，内层只定义自己的接口 |
| 一次需求变更要改5个组件 | CCP | 检查变更原因是否相同，若是→合并组件；若否→重构边界 |
| 用例直接传数据库实体到前端 | 边界规则 | 在用例层定义输出数据结构，Adapter 负责转换 |
| 项目一开始就选定了所有技术栈 | 推迟决策 | 先画边界，用接口抽象技术细节 |
| "这两个页面的代码一样，复用吧" | 真假重复 | 检查：它们会以相同原因同时变化吗？否→不要合并 |
| Main 组件里几百行业务逻辑 | Main模式 | 将业务逻辑提取到 Entity/UseCase，Main 只做组装 |
| 想拆微服务 | 解耦模式 | 先做源码/部署层次解耦，服务是物理表现不是解决方案 |
| 团队抱怨"改不动了" | 乱麻曲线诊断 | 检查变更成本趋势，识别最应该重构的边界 |
