# 整洁架构核心原则参考

> 来源：《架构整洁之道》(Clean Architecture) - Robert C. Martin
> 本文档提炼书中第1-2部分及第5部分的系统级架构原则，与 GoF 类级别设计模式形成互补。

---

## 一、架构的本质

### 1.1 设计与架构没有区别

传统的"架构=高层、设计=底层"二分法是错误的。底层设计细节和高层架构信息是**一系列决策组成的连续体**，没有清晰的分界线。软件架构的实质就是**规划如何将系统切分成组件，并安排好组件之间的排列关系，以及组件之间互相通信的方式**。

### 1.2 架构的终极目标

> 用最小的人力成本来满足构建和维护该系统的需求。

好的架构——变更成本低，且在整个系统生命周期内一直维持低成本。
坏的架构——每次发布都会提升下一次变更的成本。

### 1.3 乱麻系统的毁灭曲线

一个真实案例展示了没有架构设计的后果：

| 版本 | 工程师规模 | 生产效率 | 每行代码变更成本 |
|------|-----------|---------|----------------|
| 第1版 | 正常增长 | 正常 | 基准 |
| 第4版 | 持续增长 | 趋近于零 | 飙升 |
| 第8版 | 大规模团队 | 极低 | 第1版的**40倍** |

工程师并没有偷懒——但大部分时间消耗在对现有系统的修修补补上，而非完成新功能。

## 二、两个价值维度

每个软件系统有两个价值维度，必须同时关注：

| 维度 | 描述 | 艾森豪威尔象限 | 典型表现 |
|------|------|---------------|---------|
| **行为价值** | 软件按需求工作、产生利润 | 紧急但不重要 | 功能需求、Bug修复 |
| **架构价值** | 软件必须保持灵活、易于修改 | 重要但不紧急 | 架构设计、重构、边界划分 |

**关键逻辑链**：
- 程序可正常工作但无法修改 → 需求变更时不再能工作，也无法修改 → **价值归零**
- 程序目前不工作但易于修改 → 可以不断修改使其持续工作 → **持续产生价值**

最常见的错误：将功能需求（紧急但不重要）提到架构工作（重要但不紧急）之前。

## 三、三个编程范式——对能力的剥夺

三个编程范式都诞生于1958-1968这十年间，此后**没有新的范式出现**。

| 范式 | 核心限制（剥夺） | 架构对应 |
|------|----------------|---------|
| **结构化编程** | 限制控制权的**直接**转移（goto） | 模块的算法实现基础——创造可证伪的程序单元 |
| **面向对象编程** | 限制控制权的**间接**转移（函数指针） | **跨越架构边界的手段**——通过多态实现依赖反转 |
| **函数式编程** | 限制**赋值**语句 | 数据存放位置与访问权限的规范——隔离可变性 |

每个范式都是"剥夺"而非"赋予"能力——它们分别告诉我们**不能做什么**。

### 3.1 面向对象编程的重新定义

| 特性 | 传统认知 | 真相 |
|------|---------|------|
| **封装** | OOP提供了封装 | C语言的头文件/实现文件分离实现了**完美封装**；C++/Java反而削弱了封装 |
| **继承** | OOP独有继承 | C程序员早就通过结构体成员顺序一致的技巧实现了"继承" |
| **多态** | OOP的特性之一 | **这才是OOP的核心**——它将多态变得安全且便利，是实现插件式架构的基础 |

> 多态是跨越架构边界的根本手段。通过多态，高层策略模块定义接口，低层实现模块通过接口插入——源代码依赖方向与控制流方向**相反**。

## 四、SOLID原则的架构层面解读

> GoF skill 中已有 `solid_principles.md` 涵盖基础定义。本节只提取 Clean Architecture 在系统架构层面对 SOLID 的**重新定位**。

### 4.1 SRP：单一职责原则——按行为者划分

**传统误解**："每个模块应该只做一件事"

**真正的架构级SRP**："任何一个软件模块都应该只对**某一类行为者（Actor）**负责"

**经典案例——Employee类的灾难**：
```
class Employee {
    calculatePay()    // CFO 关心
    reportHours()     // COO 关心
    save()            // CTO 关心
}
```
这三个函数服务于三个不同的行为者，放在同一个类中：
- CFO 修改 `calculatePay` 时可能意外影响 COO 的 `reportHours`（共用 `regularHours()` 辅助函数）
- 后果：人力数据出错，给公司造成财务损失

**架构决策**：按行为者（变更原因）分离模块，而非按功能相似性。

### 4.2 OCP：开闭原则——构建变更防火墙

在系统架构层面，OCP 不仅仅是"对扩展开放，对修改关闭"，而是**整个依赖方向控制的规则体系**。

通过 SRP（按功能分组）和 DIP（依赖反转）的组合，构建出**组件层级保护体系**：
- 高层组件（Interactor/业务逻辑）对任何底层组件的变更免疫
- 所有依赖箭头指向稳定的、高层的抽象

### 4.3 LSP：里氏替换原则——系统间接口的可替换性

LSP 超越类继承，应用于**任何接口的可替换性**：

**反面案例——出租车调度系统**：
- 所有出租车公司应遵守相同的 REST 接口规范（pickupAddress、pickupTime、destination）
- 某公司将 `destination` 缩写成 `dest`
- **糟糕方案**：`if (driver.getDispatchUri().startsWith("acme.com"))` —— 具体名字入侵代码
- **正确方案**：创建调度服务抽象接口，每个公司提供自己的实现

### 4.4 ISP：接口隔离原则——避免组件间的多余依赖

架构层面的 ISP：**避免组件之间产生不必要的依赖**。

系统 S 依赖框架 F，框架 F 捆绑数据库 D。D 中包含 F 不需要的功能——这些功能的修改会导致 F 重新部署，进而迫使 S 重新部署。

### 4.5 DIP：依赖反转原则——插件架构的物理基础

**四条编码守则**：
1. 多使用抽象接口，避免依赖多变的具体实现类
2. 不要在具体实现类上创建衍生类（继承是最强的源代码依赖）
3. 不要覆盖包含具体实现的函数
4. 避免在代码中写入与任何具体实现相关的名字

DIP 是实现所有架构边界的物理机制：控制流跨越边界的**运行方向**与源代码依赖的**编译方向**正好相反。

## 五、整洁架构同心圆模型

### 5.1 四层同心圆

```
┌──────────────────────────────────────────────┐
│  外层：框架与驱动器 (Frameworks & Drivers)     │
│  ┌────────────────────────────────────────┐  │
│  │  接口适配层 (Interface Adapters)        │  │
│  │  ┌──────────────────────────────────┐  │  │
│  │  │  用例层 (Use Cases / Application) │  │  │
│  │  │  ┌──────────────────────────┐  │  │  │
│  │  │  │  实体层 (Entities)        │  │  │  │
│  │  │  │  - 关键业务逻辑         │  │  │  │
│  │  │  │  - 关键业务数据         │  │  │  │
│  │  │  └──────────────────────────┘  │  │  │
│  │  └──────────────────────────────────┘  │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

| 层 | 内容 | 变更原因 | 稳定性 |
|----|------|---------|--------|
| **实体（Entities）** | 关键业务逻辑+关键业务数据 | 核心业务规则变化 | 最稳定 |
| **用例（Use Cases）** | 特定应用的业务规则 | 应用功能变化 | 较稳定 |
| **接口适配（Adapters）** | Controller、Presenter、Gateway | 外部接口变化 | 较不稳定 |
| **框架与驱动器** | Web框架、数据库、UI、设备 | 技术选型变化 | 最不稳定 |

### 5.2 依赖关系规则（核心铁律）

> **源代码中的依赖关系必须只指向同心圆的内层。即：由低层机制指向高层策略。**

- 内层的代码**绝不能**知道外层的任何东西
- 外层可以依赖内层，内层不能依赖外层
- 穿越边界的**数据格式**应该是内层定义的简单数据结构（DTO/请求模型/响应模型），绝不能是外层的框架对象

### 5.3 跨边界数据传递规则

穿越边界传递数据时，必须遵循：
1. 数据格式应**由内层定义**，外层的 Controller/Gateway/Presenter 负责转换
2. 绝不能将数据库的行对象（如 Hibernate Entity、ActiveRecord）直接传入内层
3. 绝不能将框架的请求/响应对象传入内层
4. 不要偷懒——"反正结构一样就传过去"会破坏边界，导致架构退化

## 六、"保留可选项"——架构的终极策略

> 设计架构的策略就是：在设计中尽可能长时间地保留尽可能多的可选项。

### 6.1 什么是"可选项"

与系统**业务需求（用例）无关**的决策，都是可以推迟的可选项：
- 框架选择（Spring / Django / Express）
- 数据库选择（MySQL / PostgreSQL / MongoDB）
- Web 服务器选择
- ORM / 依赖注入框架等工具库
- 通信协议选择（REST / gRPC / GraphQL）

### 6.2 如何保留可选项——插件式架构

```
GUI ──依赖──▶ BusinessRules ◀──依赖── Database
(插件)                               (插件)
```

- 核心业务逻辑与其他组件**隔离**
- 其他组件以**插件形式**存在
- 依赖箭头由插件指向核心，**绝不反向**
- 构建起"变更无法逾越的防火墙"

### 6.3 FitNesse 的经典案例

FitNesse 团队先用 MockWikiPage 做数据访问占位，花了3个月解决 wiki 文本转换，1年后才实现持久化。最终发现文件系统就够了，根本不需要 MySQL。当客户需要 MySQL 时，仅用**一天**就实现了 MySQLWikiPage。

**教训**：通过划清边界推迟细节决策，可以节省大量时间、避免大量问题。

## 七、策略与层次

### 7.1 Level 的精确定义

> 层次（Level）= 策略与系统输入/输出之间的距离

- 离输入/输出越远 → 层次越高
- 直接管理输入/输出的策略 → 层次最低

### 7.2 层次决策规则

1. **源码依赖方向应与数据流向脱钩，与组件层次挂钩**
2. 低层组件应成为高层组件的**插件**
3. 高层策略变更不频繁但变更更重大；低层策略变更频繁但变更较小
4. 将策略隔离并统一依赖方向指向高层 → 低层紧急修改不影响高层核心组件

### 7.3 Main 组件——"最脏"的组件

Main 组件是系统中**最细节化的部分**，它的职责只有：创建、协调、监督其他组件。它依赖于系统中所有其他组件，是**依赖注入发生的地方**。Main 应该被视为最外层的插件——可以为开发/测试/生产环境分别编写不同的 Main。

## 八、测试边界

### 8.1 测试也是系统组件

测试代码和程序代码一样，是系统的一部分。设计良好的测试套件本身应遵循依赖关系规则。

### 8.2 脆弱测试问题

当测试与具体实现细节过度耦合时，即使是系统行为完全正确的重构也会导致大量测试失败。这就是**脆弱测试问题**——测试不是因为行为变化而失败，而是因为结构变化而失败。

**解决方案**：
- 测试应该测试**行为**，而不是测试实现
- 设计**测试专用 API**——封装那些会使测试脆弱的交互
- "结构性耦合"是最阴险的耦合——测试不应该知道被测试模块的内部结构
- 可测试性本身就应该是架构设计的一个维度

## 九、重复的真伪辨别

两个看起来一样的代码段：

| 类型 | 特征 | 决策 |
|------|------|------|
| **真正重复** | 每个实例的变更必须同步应用到所有副本 | **消除**——否则会引入 bug |
| **表面重复** | 两段代码走不同的演进路径（不同变更速率、不同变更原因） | **绝对不要合并**——合并后会互相拖累 |

**典型陷阱**：
- 两个用例的屏幕展现类似 → 不要复用同一段视图代码
- 数据库记录与屏幕展示数据相似 → 不要直接传数据库记录给 UI
- 按用例垂直切分时，不要因为"看起来一样"就合并

## 十、常见反模式速查

| 反模式 | 症状 | 纠正方向 |
|--------|------|---------|
| **过早决定细节** | 在写第一行业务逻辑之前就选定框架和数据库 | 推迟技术决策，用接口/Mock 占位 |
| **依赖方向外泄** | 业务层引用了框架的类/注解 | 依赖反转，定义业务层自己的接口 |
| **数据库驱动的架构** | 数据表结构决定了业务逻辑的形态 | 先设计实体和用例，数据库是"最后"的决策 |
| **框架是架构** | Spring/Django 的目录结构就是项目结构 | 核心代码应该按业务领域组织，框架代码放在外层 |
| **上帝类/Main** | Main 组件里塞满了业务逻辑 | Main 只做创建/组装，业务逻辑在 Entity/UseCase 中 |
| **假复用** | 因为"看起来一样"而合并不同变更原因的代码 | 区分真重复和表面重复 |
| **服务崇拜** | 认为"拆成微服务=好架构" | 服务边界不是架构边界，组件内部的依赖管理才是 |
