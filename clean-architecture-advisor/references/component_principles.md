# 组件构建原则参考

> 来源：《架构整洁之道》第4部分（第12-14章）
> 这是 Clean Architecture 对 GoF 最独特的超越——从类级别到部署单元级别的原则体系。

---

## 一、什么是组件

**定义**：组件是软件的**部署单元**，是整个软件系统在部署过程中可以独立完成部署的最小实体。

| 语言/平台 | 组件形式 |
|-----------|---------|
| Java | .jar 文件 |
| Ruby | .gem 文件 |
| .Net | .DLL 文件 |
| JavaScript/Node | npm 包 |
| Python | .whl / pip 包 |
| Go | module |

组件的核心价值：**可独立部署、可独立发布、可独立版本管理**。

---

## 二、组件聚合三原则

这三条原则回答了**"哪些类应该被组合到同一个组件中"**这个问题。

### 2.1 REP：复用/发布等同原则

> 软件复用的最小粒度应等同于其发布的最小粒度。

**含义**：
- 一个组件不能由毫无关联的类和模块组成
- 它们应该共享同一个主题/大方向
- 它们应该共享相同的版本号与版本跟踪
- 一个可复用的组件必须是一个可发布的组件

**架构决策**：当你要创建一个可供其他团队复用的组件时，它必须有一个完整的发布流程（版本号、发布说明、变更通知）。

### 2.2 CCP：共同闭包原则

> 将那些会**同时修改**、为**相同目的**而修改的类放到同一个组件中。

**这是 SRP 的组件版本**：
| SRP（类级别） | CCP（组件级别） |
|---|---|---|
| 一个类只对一类行为者负责 | 一个组件只对一类变更原因负责 |
| 按变更原因分离函数 | 按变更原因分离类组 |
| 共同表述：将因相同原因修改的东西放在一起 |

**架构决策**：
- 当需求变更时，该变更应尽可能只影响**一个组件**
- 如果某个需求的变更导致需要修改分布在多个组件中的代码，说明组件的聚合出了问题

### 2.3 CRP：共同复用原则

> 不要强迫一个组件的用户依赖他们不需要的东西。

**这是 ISP 的组件版本**：
| ISP（类级别） | CRP（组件级别） |
|---|---|
| 类不应依赖不使用的接口方法 | 组件用户不应依赖不使用的类 |
| 按使用者拆分接口 | 按使用者聚合组件 |

**架构决策**：
- 如果你依赖了一个组件，你会受到该组件中**所有类的所有变更**的影响
- 即使你只用了组件中的一个类，组件中其他类的bug修复/版本升级也会迫使你更新
- 因此：经常共同复用的类放在同一个组件中，不经常共同复用的类应该分开放

### 2.4 组件聚合张力图

三条原则之间存在**相互制约**的关系：

```
         CCP（可维护性——少改组件）
         /          \
        /            \
       /              \
REP（可复用性）------CRP（不依赖多余的东西）
```

| 关注点组合 | 代价 |
|-----------|------|
| 只关注 REP + CRP | 组件频繁变更，变更时需要修改大量组件 |
| 只关注 CCP + REP | 不相关的类被一起发布，用户被迫依赖不需要的东西 |
| 只关注 CCP + CRP | 组件难以被复用 |

**生命周期策略**：
- **项目早期**：CCP > REP（可维护性优先于可复用性）
- **项目成熟后**：三者需要动态平衡

## 三、组件耦合三原则

这三条原则回答了**"组件之间的依赖关系应该如何管理"**。

### 3.1 ADP：无依赖环原则

> 组件依赖关系图中**不应该出现环**。

组件依赖图必须是**有向无环图（DAG）**。

**依赖环的危害**：
- 当环中的任何一个组件发布新版本时，环中的**所有组件都必须同步发布**
- 失去了独立部署能力——而这正是组件存在的根本价值
- 环中的组件在事实上成为了一个**大组件**

**消除依赖环的方法**：

方法一——使用 DIP（依赖反转原则）：
```
原来：A ←→ B（相互依赖形成环）
解决：A → BInterface ← B
```

方法二——创建新组件：
```
原来：A ←→ B
解决：A → C ← B
```

**检测依赖环**：
- 手动检查（小型项目）
- 使用依赖分析工具（JDepend、Madge、dependency-cruiser 等）
- **每周**对组件依赖图进行一次环检测

### 3.2 SDP：稳定依赖原则

> 组件应该只依赖于比自己**更稳定**的组件。

**不稳定性度量指标**：
```
I = Fan-out / (Fan-in + Fan-out)
```

| 术语 | 含义 |
|------|------|
| **Fan-in** | 依赖该组件的组件数量（入向依赖） |
| **Fan-out** | 该组件依赖的其他组件数量（出向依赖） |
| **I = 0** | 最大稳定性——不依赖任何人，但很多人依赖它 |
| **I = 1** | 最大不稳定性——依赖很多人，但没有人依赖它 |

**依赖方向规则**：依赖箭头应该从 I 值高的组件指向 I 值低的组件。即：**不稳定的组件依赖稳定的组件**。

### 3.3 SAP：稳定抽象原则

> 一个组件的抽象化程度应该与其稳定性保持一致。

**抽象度度量**：
```
A = Na / Nc
```
- Na = 组件中抽象类/接口的数量
- Nc = 组件中类的总数量
- A = 0：完全具体（没有任何抽象）
- A = 1：完全抽象（全是接口/抽象类）

**主序列方程**：
```
D = |A + I - 1|
```
D = 到主序列的距离。D 越接近 0 越好。

### 3.4 不稳定性/抽象度坐标系

```
抽象度 A
  ↑
1.0 │    【无用区】          ★ 主序列 ★         【理想区】
    │    A=1, I=1           A+I=1             A≈1, I≈0
    │  （没人依赖的抽象）                      （稳定且抽象的）
    │
    │                         · 良好组件落在这里
    │
    │    【痛苦区】                              【痛苦区】
    │    A=0, I=0                              A=0, I=1
0.0 │  （稳定但具体——                           （不稳定且具体——
    │    数据库 Schema 等）                       OK，它们应该在这里）
    └──────────────────────────────────────────→ 不稳定性 I
    0.0                                        1.0
```

**四个区的含义**：

| 区域 | 坐标 | 含义 | 典型例子 |
|------|------|------|---------|
| **主序列** | A+I≈1 | **理想位置** | — |
| **稳定抽象区** | A≈1, I≈0 | 高度抽象且稳定 | 核心业务接口、实体层 |
| **不稳定具体区** | A≈0, I≈1 | 具体且不稳定 | 数据库驱动、UI组件、Main |
| **痛苦区** | A≈0, I≈0 | 具体但稳定——**极难修改** | 数据库 Schema、被广泛依赖的工具类 |
| **无用区** | A≈1, I≈1 | 抽象但不稳定——**没人依赖的抽象** | 没人用的过度设计接口 |

## 四、六条原则的应用框架

### 4.1 按生命周期应用

```
项目初期（原型阶段）：
  重点用 CCP（按变更原因聚合）+ SDP（稳定依赖）

项目成长期（功能迭代）：
  加入 ADP（打破依赖环）+ CRP（控制组件范围）

项目成熟期（稳定发布）：
  全六条原则 + 主序列校准（SAP）
```

### 4.2 诊断流程

```
1. 画出组件依赖图
2. 用 ADP 检查 —— 有没有环？→ 打破它
3. 用 SDP 检查 —— 有没有不稳定的组件被稳定的组件依赖？→ 反转依赖
4. 计算每个组件的 I 和 A → 标在主序列图上
5. 距离主序列很远的组件 → 考虑重构
   - 痛苦区组件 → 引入抽象接口（提高 A）
   - 无用区组件 → 删除或让它被依赖（降低 I）
6. 用 CCP 检查组件内的类 → 有没有不同变更原因的类混在一起？
7. 用 CRP 检查组件的用户 → 有没有被迫依赖不需要的类？
```

### 4.3 日常检查清单

- [ ] 组件依赖图是无环的（ADP）
- [ ] 依赖箭头从 I 高指向 I 低（SDP）
- [ ] 核心业务组件落在主序列附近（SAP）
- [ ] 每次需求变更最多只影响一个组件（CCP）
- [ ] 组件的用户不被迫依赖不需要的类（CRP）
- [ ] 组件的发布版本号与其复用的粒度匹配（REP）
