# 架构模式与边界设计参考

> 来源：《架构整洁之道》第5-6部分
> 本文档涵盖边界划分策略、解耦模式、代码组织方式，以及实现细节的分离指导。

---

## 一、边界划分策略

### 1.1 边界的本质

> 软件架构设计本身就是一门**划分边界的艺术**。
> 边界的作用是将软件分割成各种元素，以便约束边界两侧之间的依赖关系。

### 1.2 何时画线

画边界线的**核心判断标准**：

> 边界线应该画在**不相关的事情中间**，沿着系统的**变更轴**来画。

| 条件 | 判断 |
|------|------|
| GUI 与业务逻辑无关 | → 画线 |
| 数据库与业务逻辑无关 | → 画线 |
| 两个模块以不同原因、不同速率变化 | → 画线 |
| I/O 是无关紧要的（即使没有 I/O，核心逻辑仍能完成） | → 画线 |
| 两个模块总是同时变化，原因相同 | → 不画线（它们属于同一组件） |

### 1.3 画线的目的

> 在项目初期划分边界的目的是**将决策延后**，确保未来的决策不会对核心业务逻辑产生干扰。

**反面案例——P公司**：
过早决定三层分布式架构（GUI/中间件/数据库各一份领域对象），导致添加一个字段需要：
- 同步修改三层类
- 修改四个消息结构
- 修改八个传输协议处理函数

讽刺的是，P公司从未销售过需要服务器集群的系统——所有部署都在一台机器上。一个草率的架构决策将开发成本放大数倍。

## 二、边界形态速查

### 2.1 五种物理形态

| 形态 | 通信方式 | 通信成本 | 部署形式 | 适用场景 |
|------|---------|---------|---------|---------|
| **单体结构** | 同进程函数调用 | 最低（几乎免费） | 单一可执行文件 | 源码层次解耦，团队小、项目早期 |
| **部署组件** | 同进程函数调用 | 低 | 独立 .jar/.dll/.gem | 部署层次解耦，不需重新编译 |
| **线程** | 共享内存 | — | — | 非架构边界，仅执行调度 |
| **本地进程** | IPC（进程间通信） | 中等 | 独立进程 | 更明显的物理隔离 |
| **服务** | 网络通信 | 最高 | 独立主机/容器 | 最强隔离，跨网络 |

### 2.2 形态选择策略

```
项目开始 → 单体结构（源码边界）
  ↓ 团队扩大
部署组件（独立编译，共享进程）
  ↓ 需要独立扩展
本地进程/微服务（独立部署，网络通信）
```

**关键原则**：
- 通信成本越高，跨边界调用应越少
- 成本权衡：通信成本 vs 隔离收益
- 不要过早跳级——大多数系统在部署组件阶段就能满足需求

### 2.3 不完全边界

完整的架构边界很昂贵（双向多态接口、输入/输出数据结构、依赖管理），当不需要完整边界时，可以采用三种不完全边界：

| 模式 | 策略 | 适用场景 |
|------|------|---------|
| **省掉最后一步** | 定义好所有接口和组件划分，但将所有组件编译到一个部署单元中没有物理隔离 | 预期未来会分离但目前时间不允许 |
| **单向边界** | 用 Strategy 模式定义一个方向的多态接口，反向依赖通过具体类引用 | 只需要单向依赖反转的场景 |
| **门户模式（Facade）** | 不创建多态接口，只用一个外观类封装，内部直接用具体类 | 过渡期方案，边界不需要双向反转 |

YAGNI（You Aren't Gonna Need It）vs 预见性设计：找到"拐点"——当边界成本 > 当前没有边界的实际痛感时，不要画线。

## 三、三种解耦模式

| 模式 | 依赖管理 | 部署独立性 | 通信方式 | 成本 |
|------|---------|-----------|---------|------|
| **源码层次** | 代码依赖单向 | 同进程，一起部署 | 函数调用 | 最低 |
| **部署层次** | 接口定义包独立 | 独立部署，共享进程 | 函数调用 | 中等 |
| **服务层次** | 接口契约独立 | 完全独立部署 | 网络数据包 | 最高 |

**关键决策规则**：
- 一个好的架构应允许系统：单体 → 独立部署 → 微服务 → 回退到单体
- 不要一开始就做服务——做到**"一旦有需要就可以随时转变为服务"**的程度即可
- 项目早期很难知道最佳模式，最佳模式会随项目成熟度变化

## 四、业务逻辑的三层模型

### 4.1 业务实体（Entity）

关键业务逻辑 + 关键业务数据 → 绑定在一个独立软件模块内。

**特征**：
- 与数据库、UI、第三方框架**完全无关**
- 可以在任何系统中提供服务（更换整个技术栈，实体不变）
- 只包含业务逻辑，没有别的
- 位于架构的**最内层**

**例子**：银行的利息计算规则——无论用计算机还是计算器计算，规则都一样。

### 4.2 用例（Use Case）

关于如何操作自动化系统的描述，定义：
- 用户需要提供的输入数据
- 用户应得到的输出信息
- 产生输出应采取的处理步骤

**与实体的关系**：
- 用例控制实体之间的交互方式
- 用例**依赖**实体，实体不依赖用例
- 无法从用例分辨系统是 Web/命令行/服务

### 4.3 请求和响应模型

**硬性规则**：
1. 用例的输入/输出应是简单的请求/响应数据结构
2. 这些数据结构**不能**是框架对象（如 HttpRequest、HttpResponse）
3. 这些数据结构由用例层定义，适配层负责转换

## 五、展示器模式与谦卑对象

### 5.1 谦卑对象（Humble Object）模式

将难以测试的行为与易于测试的行为分离。难以测试的"谦卑对象"包含尽可能少的行为（仅做数据传递），需要测试的逻辑放在可测试的另一端。

### 5.2 展示器与视图

```
Controller → InputData
                ↓
            UseCase（Interactor）
                ↓
            OutputData → Presenter → ViewModel → View
```

- **Controller**：接收输入，转换为用例 InputData
- **UseCase（Interactor）**：执行业务逻辑，产生 OutputData
- **Presenter**：将 OutputData 转换为适合展示的 ViewModel（谦卑对象）
- **View**：接收 ViewModel 并渲染

**关键洞察**：View 不应该是可测试的对象——它只是渲染。Presenter 才是可测试的，因为它将 OutputData 转换为可断言的数据结构。

### 5.3 数据库网关

同样的谦卑对象模式应用于数据库：
- UseCase 通过**网关接口**（定义在内层）访问数据
- 数据库实现是外层的谦卑对象
- "ORM 不存在"——数据库映射代码不应出现在 Entity 或 UseCase 中

## 六、Main 组件模式

### 6.1 "最脏"的组件

Main 是系统中**最细节化的部分**，它的职责只有：
- 创建系统中所有需要的对象（依赖注入根）
- 协调、监督其他组件的运转
- 将控制权移交给系统的高层抽象部分

### 6.2 Main 作为插件

Main 应该被视为架构最外层的插件：

```
              ┌──────────┐
              │   Main   │ (插件)
              └────┬─────┘
                   ↓ 创建并注入
┌──────────────────────────────────────┐
│         核心业务逻辑（内层）            │
│         Entities + Use Cases          │
└──────────────────────────────────────┘
```

**好处**：
- 可以为不同环境（开发/测试/生产）编写不同的 Main
- 开发环境 Main：注入 Mock 数据库、本地文件系统
- 测试环境 Main：注入测试替身
- 生产环境 Main：注入真实数据库、云服务

## 七、服务的架构陷阱

### 7.1 "服务≠好架构"

微服务架构流行的理由（强隔离、独立开发部署、扩展）在实践中**未必成立**：

| 假象 | 现实 |
|------|------|
| 服务之间是强隔离的 | 如果共享数据 Schema 或通过数据库集成，就没有隔离 |
| 服务可以独立开发部署 | 如果跨服务变更需要协调发布，就不可能独立 |
| 服务可以独立扩展 | 如果数据耦合，扩展一个服务无济于事 |

### 7.2 横跨型变更

当一个需求变更需要同时修改多个服务时，架构边界实际上已经失效。真正决定架构质量的是**服务内部的组件边界**，而非服务之间的物理边界。

### 7.3 "运送猫咪"的隐喻

把一只猫关进房间，房间隔离得很好——但如果每次需要给猫喂食、清理、玩耍时都要打开房门，这隔离的意义就大打折扣。服务同理：频繁的跨服务调用（API 编排）说明服务边界画错了。

## 八、代码组织方式对比

| 方式 | 目录结构 | 适用场景 | 架构违规风险 |
|------|---------|---------|-------------|
| **按层封装** | `web/` `service/` `repo/` | 小型项目 | 高——不同业务逻辑混在一起 |
| **按功能封装** | `order/` `user/` `product/` | 中型项目 | 中——没有区分内/外代码 |
| **端口和适配器** | `domain/` `application/` `infrastructure/` | 中大型项目 | 低——内外分离 |
| **按组件封装** | `order/domain/` `order/usecase/` ... | 大型项目 | 最低——每个组件自成体系 |

**关键陷阱**：如果不使用访问修饰符（public/private/protected）来强制依赖方向，以上四种方式的区别将**完全消失**——所有类型都可以被任意引用。

## 九、实现细节隔离

### 9.1 数据库是细节

- 数据库（Oracle / MySQL / MongoDB）只是存取数据的手段，不是架构
- 应用程序的用例不应知道、不应关心数据表结构
- 业务实体不应该被 ORM 注解污染（`@Entity`, `@Table`, `@Column`）
- 数据访问代码应局限在架构最外圈

**思维实验**：假设磁盘不存在，所有数据都在内存中——我们不会用 SQL 表，而是用链表、树、哈希表。"数据库是磁盘访问的优化工具，而不是数据建模的工具。"

### 9.2 Web 是细节

- Web 只是一种 GUI，GUI 只是一个 I/O 设备
- 业务规则应该与 UI 解耦
- 架构应表现为"一个用例读取输入数据→执行处理→输出结果"——与 UI 类型无关

### 9.3 框架是细节

- 框架是"单向婚姻"——你承担所有风险，框架作者不承担任何风险
- 不要从框架基类派生核心业务类
- 不要让 `@Autowired` 等注解污染业务对象
- 业务对象应该对框架完全不知情

**安全依赖**：标准库（STL、Java 标准库）可以依赖，因为它们稳定且变更成本由语言维护者承担。
