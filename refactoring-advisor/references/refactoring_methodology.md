# 重构方法论与原则

> 来源：Martin Fowler《重构：改善既有代码的设计》第1、2、4、13、15章
> 本文档涵盖重构的定义、原则、实践方法论和哲学思想。

---

## 一、什么是重构

### 两个定义

**重构（名词）**：对软件内部结构的一种调整，目的是在不改变软件可观察行为的前提下，提高其可理解性，降低其修改成本。

**重构（动词）**：使用一系列重构手法，在不改变软件可观察行为的前提下，调整其结构。

### 两个关键约束

1. **不改变可观察行为**：重构后软件功能一如以往，用户不会感知到变化
2. **目的是可理解性和可修改性**：与性能优化形成对比——性能优化可能使代码更难理解，但出发点是执行速度

---

## 二、两顶帽子

Kent Beck 的比喻——使用重构技术开发软件时，你把时间分配给两种截然不同的行为：

| 帽子 | 行为规则 |
|------|---------|
| **添加新功能** | 不修改既有代码结构，只管添加新功能，通过测试衡量进度 |
| **重构** | 不添加功能，只改进结构，不添加测试（除非发现遗漏） |

> "软件开发过程中，你可能会发现自己经常变换帽子……整个过程或许只花十分钟，但无论何时你都应该清楚自己戴的是哪一顶帽子。"

---

## 三、为何重构

### 理由1：重构改进软件设计

- 没有重构，程序设计会逐渐**腐败变质**——人们只为短期目的修改代码
- 改进设计的重要方向是**消除重复代码**：代码越多，正确修改越困难
- "消除重复代码，就可以确定所有事物和行为在代码中只表述一次——这正是优秀设计的根本"

### 理由2：重构使软件更容易理解

- 源码的读者不仅有计算机，还有几个月后需要修改代码的程序员
- 重构帮助理解不熟悉的代码——不只是心里理解，而是动手修改让它更好地反映理解

> Ralph Johnson 的比喻：重构就像"擦掉窗户上的污垢，使你看得更远"。

### 理由3：重构帮助找到 bug

- 搞清楚程序结构的同时，也清楚了自己所做的假设

> Kent Beck："我不是个伟大的程序员，我只是个有着一些优秀习惯的好程序员。"

### 理由4：重构提高编程速度

- 良好的设计是快速开发的根本
- 恶性循环：修改时间越来越长 → 打补丁 → 新特性需要更多代码 → 速度更慢
- 重构阻止系统腐败变质，维持软件开发速度

> "良好设计是维持软件开发速度的根本。"

---

## 四、何时重构

> "几乎任何情况下我都反对专门拨出时间进行重构。重构应该随时随地进行。"

### 三次法则（Don Roberts）

> "第一次做某件事时只管去做；第二次做类似的事会产生反感，但无论如何还是可以去做；第三次再做类似的事，你就应该重构。"

**事不过三，三则重构。**

### 三个触发时机

| 时机 | 原因 |
|------|------|
| **添加功能时** | 最常见。帮助理解需要修改的代码；代码设计无法帮助轻松添加特性 |
| **修补错误时** | 收到 bug 报告就是重构信号——代码不够清晰，没清晰到一眼看出 bug |
| **复审代码时** | 不只是想象代码应该是什么样，而是"看见"它是什么样。建议可以立刻实现 |

### 间接层和重构

> "计算机科学是这样一门科学：它相信所有问题都可以通过增加一个间接层来解决。"——Dennis DeBrule

间接层的价值：
1. **允许逻辑共享**——子函数被多处调用
2. **分开解释意图和实现**——函数名解释意图，内部解释实现
3. **隔离变化**——子类可修改而不影响其他引用
4. **封装条件逻辑**——多态消息灵活清晰地表达条件逻辑

重构游戏：在保持系统现有行为的前提下，为缺乏"间接层利益"之处加入间接层。也有较少见的游戏：**找出不值得的间接层，将它拿掉**。

## 五、何时不该重构

1. **代码太混乱，重构不如重写**——"重写的一个清楚讯号就是：现有代码根本不能正常运作"
2. **项目已近最后期限**——从重构过程赢得的生产力只有在最后期限过后才能体现
3. **技术债务评估**——Ward Cunningham 的"债务"比喻：过于复杂的代码造成的额外维护成本是"利息"，应随时通过重构偿还一部分

## 六、重构与设计

**核心观点：重构不取代预先设计，它们彼此互补。**

| 没有重构时 | 有重构时 |
|-----------|---------|
| 必须保证预先设计正确无误 | 仍做预先设计，但只需"足够合理的方案" |
| 压力太大，需要更多时间和精力 | 不必找到完美方案，可以逐步改进 |

> "有了重构，你就可以通过一条不同的途径来应付变化带来的风险。你仍旧需要思考潜在的变化，但不必逐一实现所有灵活方案。应该问自己：'把一个简单的解决方案重构成这个灵活方案有多大难度？'如果答案是'相当容易'（大多数时候都如此），那么就只需实现目前的简单方案就行了。"

> "当下只管建造可运行的最简化系统，至于灵活而复杂的设计，多数时候你都不会需要它。"

## 七、重构与性能

**核心观点：重构可能使软件运行更慢，但它使软件的性能优化更容易。**

### 编写快速软件的三种方法

| 方法 | 描述 | 评价 |
|------|------|------|
| **时间预算法** | 给每个组件预分配资源预算 | 适用于性能要求极高的实时系统 |
| **持续关注法** | 任何程序员在任何时间都保持高性能 | 通常不起作用，改善分散在各处 |
| **90% 统计法（推荐）** | 先写构造良好的程序，性能优化阶段用工具找热点 | 集中优化最关键的 10% 代码 |

> "如果你对大多数程序进行分析，就会发现它把大半时间都耗费在一小半代码身上。如果你一视同仁地优化所有代码，90%的优化工作都是白费劲的。"

构造良好的程序从两方面帮助性能优化：
1. 有更多时间进行性能调整（因为能更快速地添加功能）
2. 性能分析时有较细的粒度，调整比较容易

## 八、测试——重构的前提

### 为什么测试是前提

> "如果你想进行重构，首要前提就是拥有一个可靠的测试环境。就算你有自动重构工具，你还是需要测试。"

### 自测试代码的价值

关键转变：让测试**自我检验**——屏幕上只出现 OK 或失败清单，不需要人工检查输出。

> "撰写测试代码的最有用时机是在开始编程之前。当你需要添加特性的时候，先写相应测试代码。编写测试代码其实就是在问自己：添加这个功能需要做些什么。"

预先写测试的两个好处：
1. **把注意力集中于接口而非实现**
2. **为工作安上明确的结束标志**：测试通过，工作就完成了

> "从前需要一小时甚至更多时间才能找到的错误，现在最多只需两分钟就找到了。"

### 八条测试原则

| # | 原则 |
|---|------|
| 1 | 测试必须完全自动化，让它们检查自己的结果 |
| 2 | 频繁运行测试——每天至少执行每个测试一次 |
| 3 | 收到 bug 报告先写一个单元测试来暴露这个 bug |
| 4 | 编写未臻完善的测试并实际运行，好过对完美测试的无尽等待 |
| 5 | 测试最担心的部分——这样能从测试工作中得到最大利益 |
| 6 | 寻找边界条件——第一个/最后一个/空/末尾之后 |
| 7 | 检查预期的错误——该出错时检查是否抛出了预期异常 |
| 8 | 不要因为测试无法捕捉所有 bug 就不写测试 |

### 单元测试 vs 功能测试

| 单元测试 | 功能测试 |
|---------|---------|
| 高度局部化 | 从客户角度保障质量 |
| 提高程序员生产率 | 把系统当黑箱 |
| 每个测试类隶属于单一模块 | 由独立团队开发 |
| **重构时更多倚赖** | 质量保障 |

## 九、重构的节奏（Kent Beck 的终极洞察）

### 核心信息

> "前面列出的技术仅仅是一个起点……使重构能够成功的，不是各自独立的技术，而是这种节奏。"

重构的节奏：**测试 → 小修改 → 测试 → 小修改 → 测试 → 小修改……**

### "得道"的标志

> "你又是如何得知什么时候才真正懂得这一切的呢？正是当你开始冷静下来的时候，对自己的重构技艺感到绝对自信——不论别人留下的代码多么杂乱无章，你都可以将它变好——那时你就知道，自己已经'得道'了。"

> "不过，大多数时候，'得道'的标志是：你可以自信地停止重构。"

### 停止的智慧

> "这有点像在悬崖峭壁上的小径行走：只要有光，你就可以前进，虽然谨慎却仍然自信。但是，一旦太阳下山，你就应该停止前进。"

当无法再自信地跨出下一步时：
- 代码已经比重构之前好 → 集成发布
- 没有变好 → 果断放弃，回到起点

> **"没把握就停下来。"**
> **"学习原路返回。"**

当重构失控时（连续多项修改没有测试，测试失败无法定位问题）：
- 回到最近一个没有出错的状态
- 逐一重复刚才的重构项，每次后运行所有测试
- "如果你第一次重构用了一小时，重复它只需十分钟就够了"

### 等待灵感的勇气

> "为了等待进行一项重构的后半部分所需的灵感，我最多曾经等过九个月。"

## 十一、安全重构的五个保障层次

| 层次 | 保障 | 说明 |
|------|------|------|
| 1 | 编码功力 | 程序员自己的判断力 |
| 2 | 类型检查/编译器 | 捕捉语法和类型错误 |
| 3 | 测试套件 | 捕捉编译器遗漏的行为错误 |
| 4 | 代码复审 | 捕捉以上都遗漏的设计错误 |
| 5 | 重构工具 | 程序分析确认重构是否安全可施加 |

核心安全策略：**将复杂重构定义为多个低层重构的组合**——只要每步是安全的，整个复杂重构就是安全的。

## 十三、为什么团队不重构

> 来源：Martin Fowler《重构：改善既有代码的设计》第13章

### 核心问题

为什么开发者明明知道代码需要重构，却不去做？

### 障碍分析

| 障碍 | 原因 | 解决思路 |
|------|------|---------|
| **"没时间"** | 项目进度压力大，重构被视为"额外工作" | 重构应随时随地进行，不专门拨时间；两顶帽子交替 |
| **"会破坏功能"** | 没有测试，害怕重构引入 bug | 先补测试，或用小步前进降低风险 |
| **"代码不属于我"** | 代码所有权不清，不敢改别人的代码 | 建立团队代码规范，鼓励集体 ownership |
| **"重构了也没人知道"** | 重构的价值不易量化 | 在代码审查中展示重构成果；用技术指标（复杂度、耦合度）量化 |
| **"不知道该怎么重构"** | 缺乏重构知识和经验 | 团队内部分享（如本书第1章 movie rental 示例）；建立重构文化 |
| **"管理层不理解"** | 管理层只关注功能交付，不关注代码质量 | 用业务语言解释——"重构让我们可以更快地添加新功能" |

### 推动团队重构的策略

1. **从自己代码开始**——先让自己的代码整洁，树立榜样
2. **在代码审查中展示重构**——让同事看到重构的价值
3. **用具体例子说话**——用 movie rental 这样的完整示例展示重构前后的差异
4. **建立自动化测试基础设施**——降低重构的心理门槛
5. **将重构纳入"完成"的定义**——代码整洁是功能完成的一部分
6. **庆祝消除技术债务**——让团队为代码质量提升感到自豪

> "重构的障碍往往不是技术问题，而是文化问题。改变文化比改变代码更难。"

## 十四、Movie Rental 示例：重构的节奏感

> 这是整本书的教学基石，完整展示了重构的节奏感和思考过程。

详见 `references/refactoring_example.md`。

**关键教学点**：
- 重构是小步前进的（测试 → 小改 → 测试 → 小改）
- 两顶帽子交替进行（先重构让添加功能更容易，再加功能）
- Extract Method → Move Method → Replace Conditional with Polymorphism 的完整演进路径
- "当你发现自己需要为程序添加一个特性，而代码结构使你无法很方便地达成目的，那就先重构"

## 十二、与其他 Skill 的协同

| 场景 | 使用哪个 Skill |
|------|---------------|
| "这段代码太长了/重复了/难以理解" | **本 Skill**（诊断坏味道 → 选择重构手法 → 逐步变换） |
| "重构过程中发现需要引入设计模式" | **GoF Skill**（选择 State/Strategy/Factory 等模式） |
| "系统层面需要重新组织组件和边界" | **Clean Architecture Skill**（画边界 → 管理依赖 → 隔离细节） |
| "要给重构的代码写测试" | **本 Skill** 的测试原则章节 |

### 三级工具链

```
坏代码 → [本 Skill] 诊断+修复（函数/方法/类级别变换）
         ↓ 变换中发现模式机会
       → [GoF Skill] 选择模式（类级别设计）
         ↓ 系统层面需要重组
       → [Clean Architecture Skill] 组织组件（系统级别架构）
```
